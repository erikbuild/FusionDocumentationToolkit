# Fastener Move/Copy Automation — Fusion 360 API Feasibility Research

**Date:** 2026-06-09
**Method:** 15-agent research workflow — 5 parallel research dimensions against the official Autodesk Fusion API reference, Autodesk forums, and GitHub prior art, followed by adversarial verification of the 10 highest-impact claims against primary sources. All 10 were confirmed (some with refinements, noted below).

## The Feature Under Evaluation

User selects one or more fastener component occurrences in an assembly, presses a button/hotkey, and the add-in translates them 15-20mm along their insertion axis, away from the part they are seated in — so the fastener can be screenshotted floating above its empty hole for an assembly manual. Positions must then be restorable.

## Executive Summary

**Verdict: feasible.** Every required mechanism exists in the documented add-in API and fits the toolkit's existing `register_command` pattern. No existing add-in does this; Fusion's built-in Animation-workspace auto-explode has zero API exposure and cannot scope to selected components.

Key verified mechanics:

- **Move:** `Occurrence.transform2` (read Matrix3D, add translation, assign back). Never the retired `.transform`. UI selections yield root-context occurrences whose transform2 is root/world-relative. API units are centimeters (15mm = 1.5).
- **Restore:** in parametric mode a transform2 move is "pending" and reverts unless captured via `design.snapshots.add()` — so the add-in can screenshot while pending, then `snapshots.revertPendingSnapshot()` for a zero-timeline-pollution restore. Everything inside a command execute handler is one undo transaction (single Ctrl+Z fallback).
- **Joints/grounding:** the joint solver snaps jointed occurrences back at the next solve — suppress the fastener's joints first, unsuppress to restore. `isGroundToParent` defaults true in current Fusion and silently reverts API moves; clear it before moving.
- **Axis inference (the hard part, ~70% of the effort):** joint `primaryAxisVector` as fast path (null for rigid as-built joints), clustering of cylindrical-face axes by summed area as the robust path, and a geometric test (ray cast / point containment) for the sign — no API provides "out" directly.
- **Selection:** pre-selection is readable via `ui.activeSelections` in `commandCreated`, compatible with the existing `isAutoExecute` hotkey pattern — but only for a no-dialog command; a dialog with preview is mutually exclusive with auto-execute.

Open design decisions: move vs copy semantics (the empty-hole requirement implies move); silent hotkey vs dialog-with-preview; restore mechanism; how much joint-suppression machinery v1 needs (depends on whether fasteners are typically jointed/grounded).

Items needing a quick empirical prototype before committing to a design: transform2 behavior on grounded occurrences (undocumented), `addExistingComponent` transform coordinate space, transform revert inside `executePreview`.

## Dimension 1: Moving Occurrences Programmatically

**Summary:** Moving fastener occurrences programmatically is fully feasible: Occurrence.transform2 is read/write — read the Matrix3D, add a translation (matrix.translation or transformBy), assign back — and works on nested occurrences via root-context proxies (transform2 on a proxy from root.allOccurrences is effectively world-space). In parametric mode the move is only 'pending' and reverts on the next recompute unless captured with Design.snapshots.add() (Ekins' canonical answer), which conveniently also enables a clean restore path via Snapshots.revertPendingSnapshot or by re-assigning saved matrices; in direct modeling the move just sticks. Occurrences held by Joints/AsBuiltJoints get snapped back by the joint solver, so those joints must be suppressed (isSuppressed) during the offset and unsuppressed to restore; grounded occurrences likely need isGrounded cleared (API behavior undocumented — test). Doing everything inside a CommandDefinition execute handler gives a single named undo transaction (one Ctrl+Z restores all fasteners), and MoveFeatures is not an alternative since it accepts only BRepBody/BRepFace, not occurrences.

### Findings

#### Occurrence.transform2 is a read/write Matrix3D property and is the supported way to reposition an occurrence from an add-in. The pattern is: read occ.transform2 into a Matrix3D, modify its translation (set cells (0,3),(1,3),(2,3), set matrix.translation, or compose with Matrix3D.transformBy(translationMatrix)), then assign the matrix back to occ.transform2. Brian Ekins (Autodesk API expert) demonstrates exactly this delta-move pattern in an accepted forum solution.

- **API symbols:** `adsk.fusion.Occurrence.transform2`, `adsk.fusion.Occurrence.transform`, `adsk.core.Matrix3D`, `adsk.core.Matrix3D.transformBy`, `adsk.core.Matrix3D.translation`, `adsk.core.Matrix3D.setCell`, `adsk.core.Vector3D`
- **Confidence:** high
- **Notes:** transform2 (added Jan 2022) replaces the retired transform property, which 'returns incorrect results in some cases'; Autodesk deliberately kept the old broken behavior on transform because existing programs may rely on it. Always use transform2.
- **Sources:**
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrence_transform2.htm>
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrence.htm>
  - <https://forums.autodesk.com/t5/fusion-api-and-scripts-forum/how-to-move-a-component-or-occurrence/td-p/9275882>

#### transform2 works for nested occurrences when you hold a proxy in root context. Occurrences obtained from rootComponent.allOccurrences, allOccurrencesByComponent, or UI selections are proxies whose transform2 is expressed relative to the root component (i.e., world space for the open design) — so a world-space translation can be composed directly. The transform-vs-transform2 difference only matters for deep (nested) occurrences; for direct children of root they are identical. Occurrence.createForAssemblyContext creates the proxy explicitly.

- **API symbols:** `adsk.fusion.Occurrence.transform2`, `adsk.fusion.Component.allOccurrences`, `adsk.fusion.Component.allOccurrencesByComponent`, `adsk.fusion.Occurrence.createForAssemblyContext`, `adsk.fusion.Occurrence.assemblyContext`
- **Confidence:** medium
- **Notes:** From the thread: kandennti (top community expert): 'the difference between transform and transform2 affects deep Occurrences, and there is no difference for RootComponent's child Occurrences'; the OP's accepted conclusion: transform (proxy) is 'relative to the assembly context of the Occurrence' while 'transform2 seems to return the matrix relative to the root component'. Official docs only say 'in its assembly context' and are vague for proxies — verify with a quick test on a nested fastener before committing to a design. Also note: for nested occurrences, setting transform2 moves the occurrence within its parent's space, so build the world delta then map appropriately, or simply move the top-level proxy.
- **Sources:**
  - <https://forums.autodesk.com/t5/fusion-api-and-scripts-forum/another-transformation-matrix-question/td-p/11085688>
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrence.htm>

#### In parametric mode (capture design history ON), setting transform2 moves the occurrence immediately but the move is only 'pending'. It is reverted as soon as the timeline recomputes or any other edit is made unless you capture it with Design.snapshots.add(). Ekins' accepted solution for exactly this symptom ('the component returns to the same position because you did not capture the position') is: 'Just add this call after you've transformed the occurrence. design.snapshots.add()'. Snapshots.add is only valid when there are pending transforms (Snapshots.hasPendingSnapshot / docs call it HasPendingTransforms), and each add() creates a Capture Position marker in the timeline.

- **API symbols:** `adsk.fusion.Design.snapshots`, `adsk.fusion.Snapshots.add`, `adsk.fusion.Snapshots.hasPendingSnapshot`, `adsk.fusion.Snapshots.revertPendingSnapshot`, `adsk.fusion.Snapshot.deleteMe`
- **Confidence:** high
- **Notes:** Two independent forum threads confirm the revert-without-snapshot behavior with worked examples (one moves a sphere, creates a second feature, and watches translation.x reset from -5.0 to 0.0; fix was design.snapshots.add()). For the screenshot workflow this is actually useful: you may be able to skip the snapshot entirely, take the screenshot while the move is pending, then call Snapshots.revertPendingSnapshot() ('reverts the design back to the last snapshot', valid when hasPendingSnapshot is true) as the explicit restore — no timeline pollution. In direct modeling mode (no design history) there is no timeline/capture-position concept and transform2 changes simply persist; guard snapshot calls with design.designType == ParametricDesignType.
- **Sources:**
  - <https://forums.autodesk.com/t5/fusion-api-and-scripts-forum/capture-position-after-transforming-occurance-origin/td-p/6064388>
  - <https://forums.autodesk.com/t5/fusion-360-api-and-scripts/transformation-on-occurrence-getting-reset/m-p/8402626>
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Snapshots_add.htm>
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Snapshots.htm>
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Snapshots_revertPendingSnapshot.htm>

#### Joints are not enforced at the instant you set transform2 — the occurrence moves and jointed neighbors do NOT follow — but the joint solver re-imposes joint positions on the next solve (adsk.doEvents(), recompute, snapshot capture, or subsequent edit), which snaps a fully-constrained occurrence back. The community-accepted technique for moving a jointed occurrence is to first suppress its joints (Joint.isSuppressed = True; AsBuiltJoint also has isSuppressed), set the transform, then restore. The accepted forum hack sequence was: 1) suppress joints of the part, 2) change transform of part's occurrence, 3) ground the part, 4) unsuppress the joints, 5) adsk.doEvents(), 6) unground — grounding being what holds the moved part in place while the solver runs.

- **API symbols:** `adsk.fusion.Joint.isSuppressed`, `adsk.fusion.AsBuiltJoint.isSuppressed`, `adsk.fusion.Occurrence.joints`, `adsk.fusion.Occurrence.isGrounded`, `adsk.fusion.Occurrence.isGroundToParent`, `adsk.core.Application.doEvents`
- **Confidence:** medium
- **Notes:** Occurrence.joints returns the joints affecting the occurrence (as proxies when the occurrence is a proxy) — use it to find what to suppress. Suppressing/unsuppressing a joint is itself a timeline-visible change in parametric mode. For a fastener held by a Rigid joint, a clean restore path exists: suppress joint -> transform2 offset -> screenshot -> unsuppress joint (solver snaps it back to the exact joint position). isGrounded ('pinned') and isGroundToParent are settable booleans; UI-level docs say Pin/Ground-to-Parent prevent a component from moving, but I found no primary source stating whether an API transform2 assignment on a grounded occurrence throws or silently fails — test this and clear isGrounded first if needed. Many fastener occurrences in real designs have NO joints at all, in which case transform2 + snapshot just works.
- **Sources:**
  - <https://forums.autodesk.com/t5/fusion-api-and-scripts-forum/how-to-move-a-component-that-has-joints/m-p/8884196>
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/AsBuiltJoint.htm>
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrence.htm>
  - <https://www.autodesk.com/support/technical/article/caas/sfdcarticles/sfdcarticles/Components-move-freely-even-after-grounding-subassembly-in-Fusion-360.html>

#### Undo: everything done inside a Command's execute event handler is bundled into a single transaction — one Ctrl+Z reverts it all, and the undo list shows the command's name. Official docs ('Creating Custom Fusion Commands'): 'Everything you do in the execute event handler is bundled within a single transaction and can be undone with one undo.' A plain script (or code run outside a command, e.g. directly in a button's commandCreated without execute) instead produces one undo item per API change. So an add-in that suppresses joints, sets transform2 on N fasteners, and calls snapshots.add() inside execute gives the user a single-undo restore for free.

- **API symbols:** `adsk.core.CommandDefinition`, `adsk.core.Command.execute`, `adsk.core.CommandCreatedEventHandler`, `adsk.core.Command.isPositionDependent`, `adsk.core.Command.executePreview`
- **Confidence:** high
- **Notes:** Related: Command.isPositionDependent (set in commandCreated) controls whether Fusion shows the 'Some components have been moved — Capture/Revert?' dialog when your command starts while other moves are pending (Ekins accepted solution in the move-copy thread). For an explicit 'restore' button instead of undo, store each occurrence's original Matrix3D (keyed by Occurrence.entityToken) before moving, then reassign transform2 + snapshot, or use Snapshots.revertPendingSnapshot if you never captured. Caveat: programmatic undo via Application.executeTextCommand('NuCommands.UndoCommand') style text commands is undocumented/unstable; rely on the user's Ctrl+Z or explicit transform restore.
- **Sources:**
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Commands_UM.htm>
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Command_isPositionDependent.htm>
  - <https://forums.autodesk.com/t5/fusion-360-api-and-scripts/quot-move-copy-quot-in-the-context-menu/m-p/9443651>

#### MoveFeatures is NOT an alternative for moving occurrences: MoveFeatures.createInput2's inputEntities accepts 'BRepBody or BRepFace objects but not a combination of both' (the older createInput takes an ObjectCollection of bodies). It creates a timeline Move feature that relocates geometry inside a component — it does not move the occurrence, its origin, or children, and moving a fastener's bodies would alter the component itself (affecting every other instance of that fastener component in the assembly). executeTextCommand-based moves are undocumented, version-unstable, and unnecessary here.

- **API symbols:** `adsk.fusion.MoveFeatures.createInput2`, `adsk.fusion.MoveFeatures.createInput`, `adsk.fusion.MoveFeatureInput`, `adsk.fusion.Component.features.moveFeatures`, `adsk.core.Application.executeTextCommand`
- **Confidence:** high
- **Notes:** Critical for this feature: fasteners are typically many occurrences of ONE component (e.g., 40x of 'M3x8 SHCS'). A MoveFeature on bodies would shift the bodies in ALL instances; transform2 moves exactly one occurrence, which is the behavior the documentation feature needs. Community reports also say text commands are unstable and 'changed by update without notice'.
- **Sources:**
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/MoveFeatures_createInput2.htm>
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Application_executeTextCommand.htm>
  - <https://forums.autodesk.com/t5/fusion-360-api-and-scripts/use-textcommands/td-p/9645688>

### Gotchas

- Snapshot-or-revert is mandatory in parametric mode: a transform2 move not followed by Design.snapshots.add() silently reverts on the next edit/recompute. Conversely, a snapshot adds a 'Capture Position' marker to the user's timeline — for a documentation tool that moves and restores fasteners repeatedly, consider taking the screenshot while the move is pending and calling Snapshots.revertPendingSnapshot() instead, leaving the timeline untouched; or delete the Snapshot objects you created when restoring.
- Snapshots.add() throws if there is nothing pending (docs: 'only valid when the HasPendingTransforms property returns true') — always check snapshots.hasPendingSnapshot first.
- Fastener occurrences are usually multiple instances of one component: never move bodies (MoveFeatures) or edit the component; move the specific Occurrence proxy via transform2.
- Occurrences constrained by Joints/AsBuiltJoints will be snapped back by the joint solver on the next solve; suppress the joints first (Joint.isSuppressed / AsBuiltJoint.isSuppressed via Occurrence.joints) and unsuppress to restore. Joint suppression itself appears in the timeline in parametric mode.
- Grounded/pinned occurrences (isGrounded, isGroundToParent): UI prevents moving them; whether an API transform2 assignment fails, throws, or silently no-ops on a grounded occurrence is not documented anywhere I found — must be tested empirically; safest is to temporarily clear isGrounded.
- Use transform2, never the retired transform property — transform knowingly returns wrong matrices for nested (deep) occurrences and Autodesk preserved the broken behavior for compatibility.
- For nested fasteners, operate on the root-context proxy (root.allOccurrences / UI selection / createForAssemblyContext); transform2 on that proxy is relative to the root component, so a world-space 15-20mm translation composes directly (remember API lengths are centimeters: 15mm = 1.5).
- Single-undo only works if all changes happen inside a Command's execute event handler; ad-hoc script-style code produces one undo entry per change. Set Command.isPositionDependent appropriately so Fusion's 'components have been moved' dialog doesn't interfere when the user runs other commands afterward.
- The viewport may not repaint during a long handler; adsk.doEvents() lets Fusion update the screen (and also gives the joint solver a chance to run — which can snap unsuppressed jointed parts back).
- In direct modeling (design history OFF) there is no timeline and no capture-position: transform2 moves persist immediately and snapshots are irrelevant/invalid — branch on Design.designType.
- Autodesk forums block generic fetchers (HTTP 403) but serve pages to a browser user agent; the cited forum threads were read in full that way.

## Dimension 2: Copying Component Occurrences

**Summary:** Copy semantics are feasible: Occurrences.addExistingComponent(component, transform) is the documented API equivalent of UI copy/Paste — it creates a new occurrence referencing the SAME component, landing inside whichever component's Occurrences collection you call it on. The transform's coordinate space is undocumented, and forum evidence shows the matrix is interpreted in root-component orientation even when adding into a sub-component, so the robust pattern is: resolve the parent via selectedOcc.assemblyContext, add with a root-space (or identity) matrix, then set transform2 on the root-context proxy obtained via createForAssemblyContext. For a temporary screenshot copy, adding to the root component instead of the nested parent is simpler and avoids the every-instance-duplication trap. Occurrence.deleteMe reliably removes the copy (the original keeps the component alive), and wrapping create+delete in a Command execute handler makes each step a single undo transaction.

### Findings

#### Occurrences.addExistingComponent(component, transform) creates a new occurrence referencing an existing component and is documented as 'the equivalent of copying and pasting an occurrence in the user interface' (UI Paste, shared component definition). It returns the new Occurrence or null on failure. The new occurrence lands in the component whose Occurrences collection the method is called on (e.g. rootComp.occurrences → top-level; subComp.occurrences → inside that subassembly's definition).

- **API symbols:** `adsk.fusion.Occurrences.addExistingComponent`, `adsk.fusion.Occurrence`, `adsk.core.Matrix3D`, `adsk.fusion.Component.occurrences`
- **Confidence:** high
- **Notes:** Official sample shows the canonical call: transform2 = adsk.core.Matrix3D.create(); transform2.translation = adsk.core.Vector3D.create(10, 0, 0); occ2 = allOccs.addExistingComponent(subComp1, transform2) — called on the root component's occurrences collection. API distances are in cm, so 15-20mm = 1.5-2.0 in matrix translation units.
- **Sources:**
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrences_addExistingComponent.htm>
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrences.htm>
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/ComponentSample_Sample.htm>

#### The coordinate space of the transform parameter is NOT specified in the official docs. Forum evidence (Occurrences.addNewComponent, same parameter pattern) shows that when called on a non-root component's occurrences collection, the matrix is interpreted with root-component orientation, not the sub-component's local frame — the poster had to bake the parent occurrence's rotation into the matrix to get expected placement. Treat the transform as root/world-oriented or, more robustly, add with an identity/approximate matrix and then assign the exact root-space matrix to the new occurrence's root-context proxy via transform2.

- **API symbols:** `adsk.fusion.Occurrences.addNewComponent`, `adsk.fusion.Occurrences.addExistingComponent`, `adsk.fusion.Occurrence.transform2`, `adsk.core.Matrix3D`
- **Confidence:** medium
- **Notes:** Medium confidence because the official doc is silent and the evidence is forum-observed behavior for the sibling addNewComponent method. transform2 docs say it is the position 'in its assembly context' — for a proxy in root context that means root space. Validate empirically in a test assembly before shipping.
- **Sources:**
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrences_addExistingComponent.htm>
  - <https://forums.autodesk.com/t5/fusion-api-and-scripts/matrix-of-occurrences-addnewcomponent/td-p/10821510>
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrence_transform2.htm>

#### To place the copy next to a nested original: parentOccProxy = selectedOcc.assemblyContext (docs: 'Returns the assembly occurrence ... Returns null in the case where the object is not in the context of an assembly'); if null the original is top-level and you use rootComponent. Then newOcc = parentOccProxy.component.occurrences.addExistingComponent(selectedOcc.component, m). The returned occurrence is the native object in the parent component's context; to position or select it in root context call newOcc.createForAssemblyContext(parentOccProxy) and set transform2 on that proxy. Setting transforms on non-root-context occurrences raises 'RuntimeError: 3 : transform overrides can only be set on Occurrence proxy from root component'.

- **API symbols:** `adsk.fusion.Occurrence.assemblyContext`, `adsk.fusion.Occurrence.component`, `adsk.fusion.Occurrence.sourceComponent`, `adsk.fusion.Occurrence.nativeObject`, `adsk.fusion.Occurrence.createForAssemblyContext`, `adsk.fusion.Occurrence.transform2`, `adsk.fusion.Design.rootComponent`
- **Confidence:** high
- **Notes:** Selections in root context return proxy occurrences. selectedOcc.transform2 on the root-context proxy gives the full root-space matrix of the original; multiply in the 15-20mm axis translation and assign to the copy's proxy. The proxy-only transform restriction is the single biggest correctness trap in this flow.
- **Sources:**
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrence.htm>
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/ComponentsProxies_UM.htm>
  - <https://forums.autodesk.com/t5/fusion-api-and-scripts-forum/error-while-setting-transform-in-occurrences-addnewcomponentcopy/td-p/11610437>

#### Adding the copy into a nested parent component edits that component's DEFINITION: if the subassembly component has multiple occurrences in the design, the copied fastener appears inside EVERY instance of that subassembly. For a temporary screenshot prop, adding the copy to design.rootComponent.occurrences with the original's root-space transform (selectedOcc.transform2 of the proxy, offset along the insertion axis) sidesteps this and the proxy bookkeeping entirely.

- **API symbols:** `adsk.fusion.Component.occurrences`, `adsk.fusion.Design.rootComponent`, `adsk.fusion.Occurrence.transform2`, `adsk.fusion.Occurrences.addExistingComponent`
- **Confidence:** high
- **Notes:** Follows directly from Fusion's component/occurrence data model (one definition, N instances) described in the proxies user manual. Hierarchy placement only matters for the browser tree during the screenshot; root placement is invisible in the rendered image.
- **Sources:**
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/ComponentsProxies_UM.htm>
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrence.htm>

#### Timeline (parametric mode): each occurrence creation is a timeline-tracked operation, and every state-changing API call is individually transacted — a script doing N mutations creates N undo entries, while 'Everything you do in the [Command] execute event handler is bundled within a single transaction and can be undone with one undo' (official Commands user manual). So implement the copy-out and the restore/delete each as a Command for clean single-step undo. In direct modeling mode (Design.designType = DirectDesignType) there is no timeline, and the same calls work without history entries.

- **API symbols:** `adsk.fusion.Timeline`, `adsk.fusion.TimelineObject`, `adsk.core.Command`, `adsk.fusion.Design.designType`, `adsk.fusion.Occurrences.addExistingComponent`
- **Confidence:** high
- **Notes:** API-created operations land at the current timeline marker position — make sure the design isn't rolled back before creating copies (medium confidence on marker detail, widely reported on forums). Undo can also be fired programmatically: ui.commandDefinitions.itemById('UndoCommand').execute() (forum thread 'UNDO via API', td-p/8636698).
- **Sources:**
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Commands_UM.htm>
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Timeline.htm>
  - <https://forums.autodesk.com/t5/fusion-api-and-scripts/how-do-i-quot-undo-quot-or-delete-a-failed-api-operation-aka-can/td-p/10052672>

#### The API equivalent of 'Paste New' exists: Occurrences.addNewComponentCopy(component, transform), documented as 'the equivalent of copying and using the Paste New command in the user interface' (introduced September 2022). It is NOT needed for this feature — fasteners should share the component definition (addExistingComponent), which is lighter, keeps the browser free of duplicate component definitions, and avoids addNewComponentCopy's known nested-context failure ('transform overrides can only be set on Occurrence proxy from root component'), for which the community workaround is executeTextCommand-driven Copy/FusionPasteNewCommand.

- **API symbols:** `adsk.fusion.Occurrences.addNewComponentCopy`, `adsk.fusion.Occurrences.addExistingComponent`, `adsk.core.Application.executeTextCommand`
- **Confidence:** high
- **Notes:** kandennti's workaround in that thread (Commands.Start CopyCommand / FusionPasteNewCommand / NuCommands.CommitCmd) relies on undocumented text commands — avoid unless Paste New becomes a hard requirement.
- **Sources:**
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrences_addNewComponentCopy.htm>
  - <https://forums.autodesk.com/t5/fusion-api-and-scripts-forum/error-while-setting-transform-in-occurrences-addnewcomponentcopy/td-p/11610437>

#### Occurrence.deleteMe is the documented cleanup path: 'Deletes the occurrence from the design. If this is the last occurrence referencing a specific Component, the component is also deleted.' Returns bool success. For a screenshot copy this is safe by construction — the original occurrence still references the component, so the component survives. Deleting a freshly created occurrence with no dependent features is the well-established cleanup pattern (used in forum-accepted failed-operation recovery code), and it participates in the undo transaction of the command that calls it. RemoveFeatures.add(occurrence) exists as the timeline-tracked 'Remove' alternative, but it ADDS a Remove feature to the timeline rather than erasing the creation entry, so deleteMe is the better fit for a transient copy.

- **API symbols:** `adsk.fusion.Occurrence.deleteMe`, `adsk.fusion.RemoveFeatures.add`, `adsk.fusion.RemoveFeature.itemToRemove`
- **Confidence:** high
- **Notes:** deleteMe is a hard delete; forum reports of deleteMe trouble involve occurrences/bodies with downstream timeline dependents (e.g. cuts referencing them) — not applicable to a copy created and deleted within the add-in's own session. Persist a token/attribute identifying add-in-created copies so restore can find them even after a document save.
- **Sources:**
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrence_deleteMe.htm>
  - <https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/RemoveFeatures_add.htm>
  - <https://forums.autodesk.com/t5/fusion-api-and-scripts/how-do-i-quot-undo-quot-or-delete-a-failed-api-operation-aka-can/td-p/10052672>

### Gotchas

- The transform parameter's coordinate space is undocumented for all Occurrences.add* methods; observed behavior is root-oriented even when adding into a sub-component. Always verify placement empirically; safest is add-then-set-transform2-on-root-context-proxy.
- Setting transform/transform2 on a nested occurrence that is not a root-context proxy raises 'RuntimeError: 3 : transform overrides can only be set on Occurrence proxy from root component' — use Occurrence.createForAssemblyContext(parentOccProxy) first.
- Adding the copy into a nested parent component's occurrences collection modifies the component definition: the copy appears in EVERY instance of that subassembly. Add to rootComponent for screenshot props.
- Occurrence.transform is retired ('there are cases where it returns the incorrect results') — use transform2 exclusively.
- Outside a Command execute handler, every mutating API call is a separate undo entry; wrap copy+translate (and delete) in Commands so users get single-step undo. Undo can be invoked programmatically via ui.commandDefinitions.itemById('UndoCommand').execute().
- API operations are inserted at the current timeline marker; if the user has rolled the timeline back, occurrence creation/deletion may land mid-history or fail — check Timeline.markerPosition vs count first.
- While the copy exists, the fastener's instance count is +1 (affects BOM/part-count logic the add-in or user might run during that window).
- addNewComponentCopy ('Paste New', Sept 2022) has known failures with nested-context transforms; not needed for fasteners since shared-definition instances are exactly right.
- deleteMe deletes the component too if it is the last referencing occurrence — never an issue for copies (original survives), but matters if the same code path is reused for move-semantics cleanup.
- A pasted instance does not inherit the original's joints/grounding, so it stays where the matrix puts it — desirable here, but it also means nothing constrains it if the user drags it (medium confidence, inferred from UI Paste behavior).
- In direct modeling mode there is no timeline: copy/delete leave no history, and timeline-marker logic must be skipped (Design.designType check). RemoveFeatures is a parametric-timeline feature and should not be used in direct mode.
- forums.autodesk.com blocks direct programmatic fetches (HTTP 403); thread content here was retrieved via a read proxy — re-verify quotes manually if exact wording matters.

## Dimension 3: Inferring the Insertion Axis and "Out" Direction

**Summary:** Axis inference is feasible with a two-tier strategy. Fast path: Occurrence.joints returns proxy joints affecting the selected fastener, and for non-rigid-as-built, non-inferred joints, Joint.geometryOrOriginOne/Two.primaryAxisVector gives the joint Z (the hole/screw axis) — but its coordinate space is undocumented and forum-confirmed to be component-local for linked/nested components, so transforms must be composed (or Joint.geometryOneTransform used) for nested assemblies. Robust fallback (needed because fasteners are often jointless or pinned with axis-less rigid as-built joints): cluster the fastener body's cylindrical/conical face axes (BRepFace.geometry as adsk.core.Cylinder, queried via proxies which are automatically world-space) by direction and pick the largest-summed-area direction, which survives modeled threads where single-largest-cylinder fails. The sign ('out' direction) is never provided by any API and must be computed geometrically — Component.findBRepUsingRay along plus/minus axis or BRepBody.pointContainment sampling against the seat part are the documented primitives; the larger-radius head-OD cylinder also marks the head end as a cross-check.

### Findings

#### Joint enumeration for a selected occurrence is directly supported: Occurrence.joints returns 'the joints that affect the position of this occurrence', and 'if the occurrence is a proxy, the joints returned will also be proxies in the same context as the occurrence'. Occurrence.asBuiltJoints and Occurrence.rigidGroups exist analogously. This is better than scanning Component.joints/Component.allJoints, though rootComponent.allJoints ('returns all joints in this component and any sub components ... joints in sub components will be proxies') works as a fallback for cross-checking.

- **API symbols:** `adsk.fusion.Occurrence.joints`, `adsk.fusion.Occurrence.asBuiltJoints`, `adsk.fusion.Occurrence.rigidGroups`, `adsk.fusion.Component.joints`, `adsk.fusion.Component.allJoints`, `adsk.fusion.Joint.occurrenceOne`, `adsk.fusion.Joint.occurrenceTwo`
- **Confidence:** high
- **Notes:** Joint.occurrenceOne/occurrenceTwo identify the two sides. UI convention (Fusion help + forums): the FIRST selected component is the one that moves when the joint is created, so for a fastener snapped into a hole, occurrenceOne is usually the fastener and geometryOrOriginTwo is the hole-side geometry. This is a convention, not a guarantee.

#### The insertion axis is available from joint geometry: Joint.geometryOrOriginOne/Two return JointGeometry (or JointOrigin) whose primaryAxisVector is 'the direction of the primary axis ... conceptually, this is the Z-axis' of the joint coordinate system, plus origin, secondaryAxisVector, thirdAxisVector. For a joint created on a hole's circular edge or cylindrical face, primaryAxisVector is exactly the hole/screw axis. Null cases: geometryOrOriginOne returns null for InferredJointType; AsBuiltJoint.geometry 'will return null ... in the case where the joint motion is rigid' — i.e. rigid as-built joints carry NO axis.

- **API symbols:** `adsk.fusion.Joint.geometryOrOriginOne`, `adsk.fusion.Joint.geometryOrOriginTwo`, `adsk.fusion.JointGeometry.primaryAxisVector`, `adsk.fusion.JointGeometry.origin`, `adsk.fusion.JointGeometry.secondaryAxisVector`, `adsk.fusion.JointGeometry.thirdAxisVector`, `adsk.fusion.AsBuiltJoint.geometry`, `adsk.fusion.Joint.isFlipped`
- **Confidence:** high
- **Notes:** RigidJointMotion exposes no axis vectors; only Revolute/Slider/Cylindrical motions expose rotationAxisVector/slideDirectionVector (prior art: fusion2urdf uses joint.jointMotion.rotationAxisVector / slideDirectionVector and notes they come back normalized). For joints/fasteners living directly in the root component this is moot (component space == root space). Joint.geometryOneTransform/geometryTwoTransform (Matrix3D, introduced Nov 2024, null for inferred joints) provide the joint frame even 'in cases where the JointGeometry cannot be obtained'. Since Occurrence.joints returns joint proxies in the occurrence's context, querying geometry through the proxy is the right move, but world-space correctness for nested/linked subassemblies must be verified empirically — the forum history says it was not automatic.

- **API symbols:** `adsk.fusion.Joint.geometryOneTransform`, `adsk.fusion.Joint.geometryTwoTransform`, `adsk.fusion.Joint.createForAssemblyContext`, `adsk.fusion.Joint.nativeObject`, `adsk.fusion.Occurrence.assemblyContext`, `adsk.core.Matrix3D`

#### Geometry heuristic is viable and is the necessary fallback when no joint exists: BRepFace.geometry returns an adsk.core.Cylinder for cylindrical faces, with origin, axis ('the center axis (along the length) of the cylinder'), and radius; the transient cylinder 'has no boundaries and is infinite in length', so the fastener's extent along the axis must come from the face/body bounding box. Crucially, geometry queried through proxies is automatically in root/world space: the official proxies page states querying face geometry via proxies 'will return two different points that are in world (root component) space and are the correct position. This all happens automatically.' So iterate occurrence.bRepBodies[i].faces (proxies), collect faces whose geometry is a Cylinder, and pick the dominant axis.

- **API symbols:** `adsk.fusion.BRepFace.geometry`, `adsk.core.Cylinder`, `adsk.core.Cylinder.axis`, `adsk.core.Cylinder.origin`, `adsk.fusion.BRepFace.area`, `adsk.fusion.BRepBody.createForAssemblyContext`, `adsk.fusion.Occurrence.bRepBodies`
- **Confidence:** high
- **Notes:** Largest-single-cylinder is NOT robust by itself: McMaster-style fasteners with modeled threads replace the shank cylinder with many helical/spline faces, and the head OD cylinder or a washer face can win on area. Robust variant: bucket all Cylinder/Cone face axes by direction (dot-product clustering, treating v and -v as the same bucket) and pick the direction with the largest summed area; on a screw, shank + head OD + chamfer cones are all coaxial, so the fastener axis bucket dominates regardless of threading style.
- **Sources:**

#### Component-local Z-axis convention is not a documented guarantee. Occurrence.transform2 (Matrix3D, read/write, introduced Jan 2022) 'defines this occurrences orientation and position in its assembly context' and 'replaces the transform property, which has been retired because there are cases where it returns the incorrect results' — its Z column gives the component's local Z in parent space, full world Z requires composing nested transforms. No authoritative source documents that McMaster-Carr/parts4cad fastener models are authored with the screw axis along local Z; searches found insertion how-tos but no axis convention. Use local-Z only as a tiebreaker hint, never as the primary method.

- **API symbols:** `adsk.fusion.Occurrence.transform2`, `adsk.core.Matrix3D.getAsCoordinateSystem`, `adsk.fusion.Occurrence.assemblyContext`

#### Sign determination ('which way is out') has no built-in API; ray casting is the documented primitive. Component.findBRepUsingRay(originPoint: Point3D, rayDirection: Vector3D, entityType: BRepEntityTypes, proximityTolerance=-1.0, visibleEntitiesOnly=True, hitPoints: ObjectCollection=None) -> ObjectCollection returns BRepFace/BRepEdge/BRepVertex 'arranged based on their distance from the origin point where the closest point is first'; 'the ray is infinite so the length of the vector is ignored'; each entity returned once; invisible entities skipped by default; optional hitPoints collection receives the Point3D hits. Algorithm: from the fastener's seat point (joint origin, or mid-axis), offset just beyond the fastener's own extent along +axis and -axis, cast back along the axis (or cast both directions from the joint origin and exclude the fastener's own faces by body/occurrence identity); the direction whose ray hits the seat part's faces sooner/more is 'in', so translate the opposite way. Alternative material test: BRepBody.pointContainment(point) -> PointContainment enum on the seat body, sampling points a few mm along each axis direction — simple and avoids visibility issues.

- **API symbols:** `adsk.fusion.Component.findBRepUsingRay`, `adsk.fusion.BRepEntityTypes`, `adsk.fusion.BRepBody.pointContainment`, `adsk.core.PointContainment`, `adsk.core.ObjectCollection`
- **Confidence:** high
- **Notes:** findBRepUsingRay docs do not state coordinate space or whether results are proxies; community usage indicates calling it on rootComponent with a root-space ray searches visible geometry and returns root-context entities — verify empirically (one forum answer notes selected faces come back in root context). pointContainment coordinate space follows the body you call it on: call it on the body PROXY so the point can be in world space. Sign cannot come from primaryAxisVector alone — docs never promise it points out of the hole, and Joint.isFlipped only flips relative orientation between the two sides.

### Gotchas

- JointGeometry.origin/primaryAxisVector coordinate space is undocumented; forum-confirmed that for joints involving linked/nested components the values come back in the owning component's space, not world space. Compose Occurrence.assemblyContext transforms (or use Joint.geometryOneTransform, Nov 2024+) for nested assemblies; flat root-level assemblies are unaffected.
- Rigid as-built joints — a very common way fasteners get pinned — return null from AsBuiltJoint.geometry, so they provide occurrence pairing but no axis. Inferred joints return null from Joint.geometryOrOriginOne too.
- RigidJointMotion (the typical fastener joint) exposes no axis vector; only revolute/slider/cylindrical/pin-slot motions have rotationAxisVector/slideDirectionVector. For rigid joints the axis must come from JointGeometry, not JointMotion.
- primaryAxisVector sign is not documented to point 'out of the hole'; Joint.isFlipped only controls relative orientation between the two joint frames. A geometric material test (ray cast or pointContainment sampling) is required to pick the sign.
- Largest-area single cylinder face fails on modeled-thread fasteners (thread is helical faces, not a cylinder) and can pick a washer or head OD instead of the shank. Sum areas of coaxial cylinder+cone faces per direction bucket instead.
- Component.findBRepUsingRay docs omit coordinate space and proxy behavior; community usage says call it on rootComponent with root-space rays. It honors visibility (visibleEntitiesOnly=True default) — hidden seat parts will be missed; the fastener's own faces will be the first hits and must be filtered out by body/occurrence identity.
- measureMinimumDistance returns ~0 in both axial directions for a seated fastener (it touches its seat), so it cannot determine insertion direction; use it only to verify post-move clearance.
- All API distances are centimeters (internal units); 15-20 mm = 1.5-2.0 API units. Vector3D from Cylinder.axis is not documented as normalized — call .normalize() before scaling.
- An occurrence can be affected by multiple joints (e.g., fastener jointed to two parts) or belong to a rigid group; Occurrence.joints returns all of them and the add-in must pick the joint whose geometry carries an axis, or fall back to geometry heuristics.
- Occurrence.transform2's Z column is parent-relative; for nested occurrences compose up the assemblyContext chain to get world Z. No vendor documentation guarantees fastener models are authored axis-along-Z — treat local-Z as a hint only.
- Parametric vs direct mode: joints exist in both, but in parametric (design-history) mode editing joint properties requires timeline rollTo, and moving a jointed occurrence by transform2 will be fought by the joint solver (positions revert/'Capture Position' semantics) — the axis-inference code itself is mode-independent, but test the whole move/restore flow in both modes. BRepBody.orientedMinimumBoundingBox requires Fusion builds from January 2024 onward.
- Hobbyist Voron-style assemblies frequently have fasteners placed with NO joints at all (dragged or as-built rigid), so the cylinder-face heuristic plus ray/containment sign test must be the primary path, with joints as the fast path when present.

## Dimension 4: Selection and Command UI

**Summary:** The selection/command-UI side of the explode-fasteners feature is feasible with the add-in's existing isAutoExecute pattern: the user's pre-selection is readable via ui.activeSelections inside the commandCreated handler (Autodesk's own SurfaceText sample does this), so a no-input auto-executing command can capture the picked occurrences and translate them in execute — no dialog required. If a dialog/preview flow is chosen instead, 'Occurrences' is a documented SelectionCommandInput filter string, pre-population must happen via addSelection in the activate event (explicitly invalid in commandCreated), and executePreview changes are auto-reverted because Fusion aborts the preview transaction on each re-fire and before execute. Canvas selections return proxies, so each picked entity must be resolved to its owning Occurrence via assemblyContext. There is no API-level fastener classification — selection should be 'whatever the user picked', with name/partNumber regex heuristics (McMaster part numbers, fastener-library Part Number/Description, Voron-style 'BHCS M3x8' names) or document Attributes as optional conveniences.

### Gotchas

- Pre-selection is readable in commandCreated (official SurfaceText sample does exactly this), but do not rely on it surviving later command events — stash entities or entityTokens in module state during commandCreated.
- SelectionCommandInput.addSelection is invalid in commandCreated; pre-populate in the activate event, and guard against activate firing multiple times (Pan/Orbit pre-emption re-fires it).
- Command.isAutoExecute is ignored the moment any command input exists — the dialog will show. The no-dialog hotkey flow therefore cannot have a SelectionCommandInput; it must consume ui.activeSelections captured in commandCreated.
- executePreview never fires for a no-input auto-execute command — if a live preview of the exploded position is wanted, the feature needs a dialog-style command (which conveniently also gives free auto-revert of preview transforms via the aborted transaction).
- No primary source demonstrates Occurrence.transform2 specifically inside executePreview; the documented transaction abort covers 'any creation or edit operations' but a prototype should verify transform revert, especially in parametric mode where occurrence moves involve capture-position/snapshot semantics (direct-modeling designs have no timeline and sidestep this).
- Selection.entity from canvas clicks is a BRepFace/BRepBody proxy, not an Occurrence — resolve via entity.assemblyContext; it returns None for root-component geometry and returns the TOP-LEVEL occurrence for nested paths (sub-assembly fasteners need the deepest occurrence, not the top-level one).
- Setting a SelectionCommandInput isVisible=False silently clears its selections; isEnabled currently behaves like hasFocus rather than actually disabling; addSelection permits duplicate entities (dedupe in inputChanged).

- No fastener classification exists in the API — McMaster-Carr part-number names, Fusion fastener-library Part Number/Description fields, and Voron-style names (BHCS M3x8) are all just strings; heuristics are brittle, so 'user-picked occurrences' should be the primary contract, with the Attributes API (entity.attributes.add / Design.findAttributes) for durable opt-in tagging.
- forums.autodesk.com returns HTTP 403 to plain fetchers; use a browser user agent if future research needs thread content.

**Summary:** No existing Fusion add-in (including anything by Jerome Briot) was found that programmatically explodes components or lifts fasteners for documentation shots - the closest prior art is forum-level hacks and Brian Ekins' guidance that you must move occurrences yourself via Occurrence.transform2, which is exactly the proposed feature's approach. ## Adversarial Verification of Load-Bearing Claims

The 10 highest-impact claims were independently re-checked against primary sources by verifier agents instructed to refute them. Verdicts:

### CONFIRMED: Occurrence.transform2 is a read/write Matrix3D property and is the supported way to reposition an occurrence from an add...

**Refined claim:** Occurrence.transform2 is a read/write Matrix3D property and is the supported way to reposition an occurrence from an add-in. The pattern — read occ.transform2 into a Matrix3D, modify its translation (setCell on (0,3),(1,3),(2,3), set matrix.translation, or compose via transformBy), then assign back to occ.transform2 — is correct; all those Matrix3D members exist. transform2 (added January 2022) replaces the retired transform property, whose incorrect-in-some-cases behavior Autodesk deliberately preserved for existing programs. Refinements: (1) In the cited forum thread, Brian Ekins does demonstrate exactly this delta-move pattern (getCell+setCell on the translation cells, assign back), but his post is NOT the accepted solution — the accepted answer is the original poster's own code — and Ekins's example is C++ using the old `transform` property, since the 2020 thread predates transform2. (2) Practical caveats: assigning transform2 can silently fail to stick — occurrences with isGroundToParent true (the default since mid-2024) revert to their original position (set occ.isGroundToParent = False first, per Ekins), joints/grounding can snap occurrences back, and in parametric designs the move is positional only unless captured with design.snapshots.add() (capture position).

**Evidence:** OFFICIAL DOCS (primary sources, verified directly):
1. https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrence_transform2.htm — Occurrence.transform2 is a read/write property returning adsk.core.Matrix3D ("Gets and sets the 3d matrix data that defines this occurrences orientation and position in its assembly context"), introduced January 2022, and "replaces the transform property, which has been retired because there are cases where it returns the incorrect results."
2. https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrence_transform.htm — confirms the retired property note nearly verbatim to the claim: "This method remains and has the same behavior as before. It returns incorrect results in some cases but some existing programs may be relying on these bad results so this property remains unchanged."
3. https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Matrix3D.htm — Matrix3D has a read/write `translation` property, `transformBy()`, `setCell()`, and `getCell()`, so all three claimed ways of modifying the translation exist.

FORUM THREAD (fetched raw HTML of https://forums.autodesk.com/t5/fusion-api-and-scripts-forum/how-to-move-a-component-or-occurrence/td-p/9275882): Brian Ekins (BrianEkins) does post code in this 2020 thread demonstrating exactly the read-modify-assign pattern with both absolute and delta-move modes: `Ptr<Matrix3D> trans = occ->transform(); ... trans->setCell(0, 3, trans->getCell(0, 3) + delta->x()); ... occ->transform(trans);`. HOWEVER, two attribution errors in the claim: (a) Ekins's code post (message 9278347) is NOT the accepted solution — the accepted solution (message 9278111) is by the original poster RogerInHawaii, who wrote his own helper methods; Ekins's posts are regular replies. (b) The thread predates transform2: Ekins's example is C++ and uses the old `transform` property, not `transform2`.

REAL-WORLD CAVEATS (forum evidence): https://forums.autodesk.com/t5/fusion-api-and-scripts/malfunctions-with-the-latest-update-of-fusion/td-p/12824772 — a 2024 thread where a component moved via transform2 reverts to its original position as soon as the user interacts with it; Brian Ekins explains the cause is the then-new "Ground to Parent" default and recommends `if occ.isGroundToParent: occ.isGroundToParent = False` before repositioning. Additionally, forum code examples (e.g. https://forums.autodesk.com/t5/fusion-api-and-scripts-forum/getting-point-location-after-transform/td-p/12089889) pair transform2 with `design.snapshots.add()` to capture position in parametric designs; in parametric mode an uncaptured move is positional only, and occurrences constrained by joints or grounding will snap back.

### CONFIRMED: transform2 works for nested occurrences when you hold a proxy in root context. Occurrences obtained from rootComponent.a...

**Refined claim:** transform2 works for nested occurrences when you hold a root-context occurrence. Nested occurrences obtained from rootComponent.allOccurrences, allOccurrencesByComponent, or UI selections are proxies in root context; direct children of root from those same sources are native occurrences, not proxies (assemblyContext returns null). In root context, transform2 is root-relative (world space for the open design) per community testing (kandennti, forum thread 11085688) — official docs only say "in its assembly context." The transform-vs-transform2 difference matters only for deep (nested) occurrences; for direct children of root they are identical (kandennti). Occurrence.createForAssemblyContext creates a proxy explicitly. Use transform2 unconditionally: transform is officially retired (docs: retired July 2025) and returns incorrect results in some cases. Caveats: (1) the getter's root-relativity for proxies is community-verified, not documented, and the setter's frame for nested occurrences is undocumented — test on a nested occurrence before relying on it; (2) joints constrain components, not occurrences, so jointed occurrences can read identity transforms and may snap back when moved; (3) in parametric designs, transform changes are pending until captured with Design.snapshots.add() (valid when hasPendingTransforms is true) and can revert on recompute/reopen without a snapshot; direct-modeling designs have no timeline/snapshot requirement.

**Evidence:** 1. Official API reference (https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrence.htm): All claimed symbols exist. Occurrence.transform2: "Gets and sets the 3d matrix data that defines this occurrences orientation and position in its assembly context. This property replaces the transform property, which has been retired because there are cases where it returns the incorrect results." transform2 property page (https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrence_transform2.htm) — read/write Matrix3D, introduced January 2022. Occurrence.transform page (https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrence_transform.htm) — RETIRED (introduced Aug 2014, retired July 2025); behavior left unchanged because existing programs may rely on its bad results. createForAssemblyContext: "Creates or returns a proxy for the native object"; assemblyContext returns null for native objects. Component.allOccurrences (https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Component_allOccurrences.htm) and allOccurrencesByComponent (https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Component_allOccurrencesByComponent.htm) exist as claimed; note the docs say "in its assembly context" and never explicitly state root-relativity for proxies — that part rests on community testing, as the claim's notes acknowledge.

2. Cited forum thread verified (https://forums.autodesk.com/t5/fusion-api-and-scripts-forum/another-transformation-matrix-question/td-p/11085688, retrieved via text proxy after direct fetch was 403): kandennti's quote is accurate — "As for the difference between transform and transform2, it affects deep Occurrences, and there is no difference for RootComponent's child Occurrences." OP's accepted conclusion matches: transform is relative to the occurrence's assembly context, transform2 relative to the root component. The same thread adds a practical caveat the claim should carry: rigid joints constrain components (not occurrences), so a jointed occurrence's transform can read as identity even when visually positioned elsewhere.

3. Primary source on proxies/root context — Brian Ekins (Fusion API designer), AU2017 class SD125353 "Components, Components, Components" (https://forums.autodesk.com/autodesk/attachments/autodesk/fusion-api-scripts-forum-en/8713/1/SD125353-Ekins-AU2017.pdf), p18: "selections return proxies that are relative to the root component"; p20: "When interactively selecting entities, they're always proxies to the real entity unless the entity is in the root component." ### CONFIRMED: Adding the copy into a nested parent component edits that component's DEFINITION: if the subassembly component has multi...

**Refined claim:** Adding a copy into a nested parent component edits that component's DEFINITION: child occurrences belong to the component (Component.occurrences), so if the subassembly has multiple occurrences in the design, the copied fastener appears inside EVERY instance ("Any changes made to one instance of the component are also made to all other instances" — ComponentsProxies_UM). For a temporary screenshot prop, design.rootComponent.occurrences.addExistingComponent(fastenerOcc.component, matrix) sidesteps this and the proxy bookkeeping: it creates one extra instance of the same component definition at root. The placement matrix is root/model space, and selectedOcc.transform2 supplies it ONLY because UI selections return occurrence proxies in root context — transform2 is "in its assembly context," so on a native nested occurrence it is parent-relative, and setting it there raises "transform overrides can only be set on Occurrence proxy from root component." The insertion-axis offset must be expressed in root space and is best baked into the Matrix3D passed to addExistingComponent. Two practice caveats: since the May 2024 release new occurrences default to isGroundToParent = True, which snaps back any post-creation transform2 move (set it False first); and in parametric designs un-captured occurrence moves are pending transforms that a recompute can revert unless design.snapshots.add() is called — both moot if the prop is placed once at creation and deleted right after the screenshot.

**Evidence:** All four API symbols exist in the official Fusion API reference, and the data-model premise is directly supported by Autodesk's own user manual.

1. Definition-vs-instance premise (CONFIRMED). The Components/Proxies user manual (https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/ComponentsProxies_UM.htm) states: "An occurrence can be thought of as an instance of a component" and "Any changes made to one instance of the component are also made to all other instances (occurrences)... Edits made to the component affect both occurrences." Component.occurrences (https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Component_occurrences.htm) is a read-only property returning "the occurrences at the top-level of this component" — i.e., child occurrences are part of the component definition, so adding a fastener occurrence inside a nested component replicates it into every instance of that subassembly.

2. Symbols (CONFIRMED). Design.rootComponent exists, read-only, returns the root Component (https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Design_rootComponent.htm). Occurrences.addExistingComponent(component, transform) exists, is "the equivalent of copying and pasting an occurrence in the user interface," and the Matrix3D "defines the location for the new occurrence" (https://help.autodesk.com/cloudhelp/ENU/Fusion-360-API/files/Occurrences_addExistingComponent.htm) — note it creates a new instance of the SAME component definition, which is exactly what the claim wants. Occurrence.transform2 exists: "Gets and sets the 3d matrix data that defines this occurrences orientation and position in its assembly context. For a UI-selected occurrence — which is always a proxy ("When working interactively in an assembly and selecting geometry, everything is a proxy," ComponentsProxies_UM) with full root path — the assembly context is root, so transform2 is the root-space transform. ### CONFIRMED: Timeline (parametric mode): each occurrence creation is a timeline-tracked operation, and every state-changing API call ...

**Refined claim:** In parametric designs (Design.designType = ParametricDesignType), occurrence creation via adsk.fusion.Occurrences.addExistingComponent ("equivalent of copying and pasting an occurrence in the user interface") is timeline-tracked — each Occurrence exposes a timelineObject for its creation. Per the official Commands user manual, "every API call that causes a change within Fusion will show up as a separate operation in the undo list," so a script doing N mutations outside a command creates N undo entries, while "everything you do in the execute event handler is bundled within a single transaction and can be undone with one undo." Therefore implement copy-out and restore/delete each as an adsk.core.Command (scripts may also create and execute a CommandDefinition themselves, but must not auto-terminate so they can handle command events). In direct modeling mode (DirectDesignType) there is no timeline and "further operations will not be captured in the timeline." Caveats: the "API operations land at the current timeline marker" detail is forum-reported, not officially documented — avoid creating copies while the design is rolled back, since Fusion may relocate or reject items. For restoring occurrence positions, use Occurrence.transform2 (transform is retired), and note that in parametric mode occurrence moves are not timeline-captured and can revert on recompute unless a snapshot is created. Programmatic undo via ui.commandDefinitions.itemById('UndoCommand').execute() is confirmed (forum thread "UNDO via API", td-p/8636698, under /fusion-360-api-and-scripts/).

**Evidence:** All five API symbols exist and the core transaction claim is confirmed by primary sources:
